Help Documentation

Learn how to use StackWatch monitoring features

Getting Started

StackWatch is a comprehensive server monitoring and infrastructure management platform. This guide will help you get started with the key features.

Adding a Server

To start monitoring a server:

  • Navigate to "Add Server" from the sidebar or server list
  • Enter the server's name, IP address, SSH port, username, and password
  • The system will automatically deploy SSH keys for secure, passwordless access
  • Required dependencies (psutil) will be installed on the remote server
  • Initial metrics will be collected immediately after setup
Note: The password is used only once for initial SSH key deployment and is never stored. All subsequent connections use SSH key authentication.

Viewing the Dashboard

The main dashboard provides an overview of your infrastructure:

  • Reliability Metrics: System uptime and SLI/SLO compliance
  • CPU & Memory Trends: Real-time charts with spike detection (triangle markers)
  • Disk I/O Summary: Read/write operations across servers
  • Health Status: Quick view of online, warning, and offline servers
  • Recent Alerts: Timeline of recent alert events
  • Trend Insights: AI-analyzed patterns and recommendations
  • AI Recommendations: Intelligent suggestions for optimization

Server Details

Click on any server from the dashboard or server list to view detailed metrics:

  • Real-time CPU, memory, disk, and network usage
  • Top processes consuming CPU and memory
  • Service status and monitoring controls
  • Historical metrics with customizable time ranges (1h, 6h, 24h, 7d)
  • Anomaly detection results and AI explanations
  • Disk usage by partition with type detection (SSD/HDD/NVMe)
  • System uptime and load averages

Monitoring Features

Server Status

The system continuously monitors server connectivity through SSH heartbeat checks:

  • Online: Server is responding and has no active alerts
  • Warning: Server is online but has active alerts or anomalies
  • Offline: Server is not responding or monitoring is suspended

Heartbeat checks run every 30-60 seconds to ensure accurate status reporting.

Alerts & Notifications

Configure alerts to be notified when issues are detected:

  • Email Notifications: Configure SMTP settings (Gmail, Outlook, custom SMTP)
  • Slack Notifications: Set up Slack webhook for instant team alerts
  • Threshold Alerts: CPU, memory, disk usage, and I/O thresholds
  • Service Alerts: Get notified when monitored services go down
  • Connection Alerts: Alerts when servers go offline or come back online
New: CPU and Memory alerts now capture the top processes at the time of the spike, helping you identify the cause of resource issues.

Anomaly Detection

The system automatically detects anomalies in your server metrics:

  • Uses ADTK (Anomaly Detection Toolkit) and machine learning algorithms
  • Detects threshold violations, level shifts, and volatility changes
  • Severity levels: Low, Medium, High, Critical
  • AI-powered explanations for detected anomalies
  • View and resolve anomalies from the server details page

Service Monitoring

Monitor critical services running on your servers:

  • Auto-discovery of running services (Apache, Nginx, MySQL, PostgreSQL, etc.)
  • Enable/disable monitoring for individual services
  • Real-time service status (running/stopped/failed)
  • Service latency measurements for web services
  • Alerts when services go down or come back up

Log Troubleshooting

Log Monitoring

Monitor and analyze log files from your servers:

  • Configure log paths to monitor (e.g., /var/log/apache2/error.log)
  • Automatic error detection and categorization
  • View log events with source IP addresses
  • Similar logs are grouped together with IP counts
  • AI-powered solutions for common errors
  • Save solutions for future reference

AI Solutions

Get AI-powered help for troubleshooting:

  • Click "Ask AI for Solution" on any log error
  • AI analyzes the error context and provides recommendations
  • Solutions include step-by-step instructions
  • Save useful solutions for your team
  • View saved solutions in the "Saved Solutions" tab

Server Management

  • Edit Server
    Update server name, IP address, and SSH port. Changes take effect immediately.
  • Suspend Monitoring
    Temporarily pause monitoring, data collection, and alerts for a server without removing it from the system. Server will show as "Offline" while suspended.
  • Suppress Alerts
    Stop receiving alert notifications while keeping monitoring active. Useful for planned maintenance windows.
  • Configure Thresholds
    Set custom alert thresholds for CPU, memory, disk usage, and I/O metrics per server.
  • Manage Services
    Enable or disable monitoring for specific services. View service status and configure alert settings.
  • Select Monitored Disks
    Choose which disk partitions to monitor for space alerts. Exclude virtual filesystems automatically.
  • Delete Server
    Remove a server from monitoring. This action cannot be undone and will delete all associated metrics and alerts.

Alert Configuration

Email Configuration

Set up email alerts:

  • Gmail: Use smtp.gmail.com with an App Password (requires 2FA)
  • Outlook/Office 365: Use smtp.office365.com with your credentials
  • Custom SMTP: Configure any SMTP server with TLS/SSL options
  • Test your configuration before saving
  • Multiple recipients supported (comma-separated)

Slack Configuration

Set up Slack notifications:

  • Create an Incoming Webhook in your Slack workspace
  • Paste the webhook URL in the configuration
  • Optionally specify a channel, username, and icon
  • Test the configuration to verify it works
  • Alerts will be sent to Slack alongside email (if both configured)

User & Role Management

Managing Users

Create and manage user accounts:

  • Create new users with specific roles and permissions
  • Edit user information, passwords, and access levels
  • Activate or deactivate user accounts
  • Assign roles to control what users can access
  • Superusers have full access to all features

Roles & Permissions

Control access with role-based permissions:

  • Create custom roles with specific privileges
  • Available privileges: view servers, manage servers, manage users, configure alerts, etc.
  • Edit or delete roles (except protected system roles)
  • View which users are assigned to each role

Settings

  • Timezone
    Set the display timezone for all timestamps in the application. All data is stored in UTC and converted for display.
  • Language
    Choose the interface language for the application.
  • Log Retention
    Configure how long log events are retained (7, 15, or 30 days). Older logs are automatically purged.

Data Collection & Security

How Data is Collected

StackWatch collects metrics using read-only operations:

  • A lightweight Python script is uploaded to /tmp/collect_metrics.py on each monitored server
  • The script uses psutil to read system metrics (CPU, memory, disk, network)
  • No data is modified on client servers - all operations are read-only
  • Connections are made via SSH using key-based authentication
  • Metrics are collected every 1-5 minutes depending on server load

Security Practices

StackWatch follows security best practices:

  • Server passwords are never stored - only used once for SSH key deployment
  • SSH key authentication is used for all subsequent connections
  • No destructive operations are performed on monitored servers
  • The monitoring script is read-only (no service restarts, no file modifications)
  • All communication is encrypted via SSH

Best Practices

  • Set Appropriate Thresholds
    Configure alert thresholds based on your server's normal usage patterns. Start with defaults (80% CPU, 80% memory, 90% disk) and adjust based on observed behavior.
  • Monitor Critical Services
    Enable monitoring for services that are critical to your application (web servers, databases, caches). This ensures immediate notification when they go down.
  • Review AI Recommendations
    Check the AI Recommendations panel regularly for optimization suggestions. The AI analyzes your metrics over time to identify potential improvements.
  • Use Alert Suppression for Maintenance
    Before performing planned maintenance, enable alert suppression to avoid unnecessary notifications. Remember to re-enable alerts afterward.
  • Configure Both Email and Slack
    For critical infrastructure, configure both email and Slack notifications to ensure alerts are received through multiple channels.
  • Review Alert History
    Regularly review the Alert History page to understand patterns, identify recurring issues, and verify that alerts are being resolved.

Troubleshooting

  • Server Shows Offline
    Check that the server is reachable via SSH. Verify the IP address, port, and that the SSH key is still authorized on the server. Check if monitoring is suspended.
  • No Metrics Being Collected
    Ensure psutil is installed on the monitored server (pip3 install psutil). Check SSH connectivity and verify the monitoring cron jobs are running.
  • Email Alerts Not Sending
    Test your email configuration using the "Test" button. For Gmail, ensure you're using an App Password (not your regular password). Check spam folders.
  • Slack Alerts Not Working
    Verify the webhook URL is correct and the Slack app is installed in your workspace. Test the configuration using the "Test" button.
  • High CPU/Memory But No Alert
    Check if alert suppression is enabled for the server. Verify the threshold values in the server's monitoring configuration.
