Help & Documentation

Stack Alert is a push-based infrastructure monitoring platform. A lightweight agent on each server dials out to Stack Alert over HTTPS — there are no inbound ports to open and no credentials to your fleet stored here, so it works behind NAT and in air-gapped networks. This guide covers everything from installing the agent to alert routing, roles and licensing.

1 Getting Started

Add a server by installing the agent on it, then watch metrics, alerts and anomalies flow into the dashboard.

Adding a server (install the agent)

Stack Alert never logs in to your servers. Instead you install a small agent on each one:

  • Go to Servers → Add Server and enter a name, IP and OS type (Linux or Windows).
  • Stack Alert creates the server and shows a ready-to-run install command carrying a unique, per-server token.
  • Run that command on the target machine — install.sh on Linux, install.ps1 (or the packaged .exe) on Windows. It installs the agent as a service and starts it.
  • The agent immediately begins pushing metrics; the server turns Online within a minute or two.
No SSH, no passwords. The agent authenticates with its own bearer token and connects outbound only over HTTPS (443). Nothing needs to be opened toward the server.

The dashboard

The dashboard is your fleet at a glance:

  • Health banner — how many targets are healthy vs need attention, refreshed live.
  • Top consumers — the busiest servers by CPU, memory and disk.
  • CPU / Memory trends — charts with Avg/Min/Max, an average line, a top-process tooltip and a pop-out.
  • Header icons — the ✨ (Trends & AI) and 🔔 (Anomalies) icons open as modals over the page.
  • Persona toggle — Admins/CEOs can switch between the Operations and Executive views (top right).

Server details

Click any server to drill in:

  • Live CPU, memory, disk and network, with historical ranges (1h / 6h / 24h / 7d).
  • Top CPU and memory processes.
  • Services and containers, with per-item monitoring toggles.
  • Disk usage per partition (with SSD/HDD/NVMe detection) and system uptime / load.
  • Anomalies detected for this server, with optional AI explanations.

2 How monitoring works

The push-agent model

  • Each monitored server runs the Stack Alert agent, which dials out to this server over HTTPS and authenticates with a per-server bearer token (only a hash of the token is stored here).
  • Because it is outbound-only, you don't open inbound firewall rules or expose the servers — it is NAT- and air-gap-friendly.
  • A compromise of the Stack Alert box gives an attacker no way in to your fleet: this server holds no credentials that can log into any monitored machine.
  • The agent runs as a background service (systemd on Linux, a Windows service) and restarts automatically.

Server status & heartbeats

The agent sends regular heartbeats and metric pushes. Status is derived from that:

  • Online — reporting normally, no active issues.
  • Warning — online but with active alerts or anomalies.
  • Offline — not reporting (agent stopped / host down / lost connectivity), or monitoring is suspended.

3 Monitoring features

Anomaly detection

Stack Alert learns each server's normal behaviour and flags deviations automatically:

  • Statistical + ML detection (ADTK / Isolation Forest) for spikes, level shifts and volatility.
  • Severity levels: Low, Medium, High, Critical. High/Critical also raise alerts.
  • Per-server sensitivity (or Off) in the server's monitoring configuration.
  • Memory-leak detection (shared-memory / IPC / process RSS growth) is surfaced as a Capacity signal.
  • When the optional local AI is enabled, each anomaly gets a plain-English Cause / Fix explanation.

Services & containers

  • Auto-discovery of running services and Docker containers.
  • Enable/disable monitoring per service or container.
  • Real-time up/down state, with alerts when a monitored unit goes down or recovers.

Synthetic uptime & security

  • Synthetic checks probe URLs/endpoints on a schedule and raise Availability alerts on down/recovery.
  • Security monitoring surfaces events such as failed SSH logins on the Security page and raises Security alerts.

4 Alerts & routing

Every alert belongs to one category and carries a severity. Routing decides who is notified — separately for email and Slack.

Resource Availability Security Capacity Business

Email routing (by role & severity)

On Alert Configuration → Alert Routing, each cell of the Role × Category matrix sets the minimum severity at which that role is emailed (or Off). Recipients are the email addresses of the active users who hold a qualifying role — there is no single "to" address any more.

  • Hover a category name for a tooltip explaining what feeds it.
  • Example: set CEO → Availability = Critical so leadership is only paged when a server is down.
  • Editing the matrix is Admin-only (it is cross-role policy).
Per-user mute: in Users → Edit user, turn off Email alerts to exclude that person from all alert emails regardless of their role.

Slack routing (by category & severity)

Slack uses a single shared webhook, so there is no per-user dimension. On the Slack tab, the Slack Routing table sets a minimum severity per category (or Off). An alert posts to Slack only if Slack is enabled and its category/severity passes the rule.

Setting up channels

  • Email: pick a provider (Gmail via App Password, Outlook/Office 365, or custom SMTP) and enter the account; alerts are sent from that authenticated address. Use Save Configuration and the Test button.
  • Slack: create an Incoming Webhook, paste the URL, optionally set a channel/username/emoji, and enable it.

Alerts History

  • Review every alert, filter by acknowledged/unacknowledged, category and time range.
  • Acknowledge alerts (individually or in bulk) — acknowledged alerts can't be re-acknowledged.
  • There is no manual delete: alert data is pruned automatically by the Data retention setting.

5 Server management

  • Edit server
    Update the server's name, IP and details. Changes take effect immediately.
  • Suspend monitoring
    Pause data collection and alerts without removing the server. It shows as Offline while suspended.
  • Suppress alerts
    Keep collecting data but stop notifications — ideal for planned maintenance.
  • Thresholds
    Set per-server CPU, memory, disk and I/O thresholds that trigger Resource alerts.
  • Services & disks
    Choose which services/containers and which disk partitions to monitor.
  • Remove server
    Removal shows the OS-specific agent-uninstall steps and asks you to type the server name to confirm. This deletes its metrics and alerts.
Re-deploying an agent: installing again on the same machine refreshes the token and restarts the service, so a re-issued server comes back Online automatically.

6 Users, roles & access

Roles

Access is role-based and enforced on the server (the UI only reflects what the server already permits):

  • Admin — full access, including users, roles, alert routing and licensing.
  • Operator — day-to-day operations (servers, alerts, monitoring), no user/role admin.
  • CEO — the Executive view and business signals; a read-slanted leadership persona.
  • Create custom roles on the Roles page and assign capabilities.

Personas & impersonation

  • Personas: the dashboard has an Operations and an Executive view; users land on the one that fits their role and can switch if permitted.
  • Impersonation: Admins/CEOs can "view as" a lower-privilege user to see exactly what they see — actions are audited and the impersonator can never gain extra privileges.
  • Email alerts can be toggled per user (see Alerts & routing).

7 Executive view & reports

  • Executive dashboard — right-sizing recommendations that compare each VM's real usage against its allocation and suggest scaling up/down.
  • Operations report (/reports/operations/) — an operational summary of fleet health and alerts.
  • Executive report (/reports/executive/) — a leadership-oriented summary for sharing.

8 Licensing

Stack Alert uses an offline, signed license — no phone-home, so it works air-gapped. Open License (under Administration):

  • Editions: Standard and Pro (Pro unlocks Windows agents, the Executive dashboard, AI insights, Security and Business).
  • Server cap & expiry: each license allows a number of servers and has a renewal date.
  • Install ID: shown on the License page — send it to your vendor to have a license issued/renewed, then paste the license string back in.
  • Evaluation mode: with no license installed, all features are unlocked for evaluation (an admin-only banner reminds you).
After expiry there is a grace period; beyond that the app becomes read-only (data keeps flowing in) until a renewed license is installed.

9 Settings

  • Display timezone
    All timestamps (metrics, alerts, logs, charts) display in this timezone. Data is stored in UTC and converted at display time.
  • Language
    Interface language for the application.
  • Data retention
    Keep the last N days (7–365) of collected data — metrics, incidents, alerts and logs. Older data is pruned automatically each day; daily summaries are kept longer for long-range trends.

10 Security & privacy

  • Outbound-only agents — no inbound ports on your servers; nothing to expose.
  • Token auth — each agent has its own bearer token; only a hash is stored here, and it grants nothing beyond pushing that server's data.
  • No fleet credentials — this server can't log into any monitored machine, so compromising it doesn't compromise your fleet.
  • Read-only collection — the agent reads system metrics; it doesn't restart services or modify the host.
  • Offline licensing — verified locally against an embedded public key, safe for air-gapped installs.

Best practices

  • Tune thresholds & sensitivity
    Start with defaults, then adjust per-server CPU/memory/disk thresholds and anomaly sensitivity to match each workload's normal.
  • Route deliberately
    Use the routing matrix so each role gets the right categories at the right severity — avoid alert fatigue by raising the minimum severity for noisy categories.
  • Suppress during maintenance
    Turn on alert suppression before planned work, and remember to turn it back off.
  • Use both channels for critical infra
    Configure email and Slack so critical alerts reach people through more than one path.
  • Right-size regularly
    Check the Executive view's recommendations to reclaim over-provisioned VMs or relieve strained ones.

Troubleshooting

  • Server shows Offline
    Check the agent service is running on the machine (systemd on Linux, Services on Windows) and that it can reach Stack Alert outbound on 443. Confirm monitoring isn't suspended. Re-running the installer refreshes the token and restarts the agent.
  • No metrics coming in
    Verify the agent started without errors and the server's token matches (re-install to re-issue). A proxy/firewall blocking outbound HTTPS is the most common cause.
  • Someone isn't getting email alerts
    Check the Alert Routing matrix (their role + category must be at or below the alert's severity, not Off), confirm their user has Email alerts enabled, and that they have an email address. Use the SMTP Test button.
  • Slack not posting
    Confirm Slack is enabled, the webhook URL is valid, and the alert's category passes the Slack Routing minimum severity. Use the Slack Test button.
  • App is read-only
    The license has expired past its grace period. Install a renewed license on the License page — ingest keeps running throughout.
  • High usage but no alert
    Check the server's thresholds and whether alert suppression is on, and confirm the Resource category is routed to a role/severity that reaches you.
